# AI Chief of Staff -- Agent 架构提案

**Author**: Alex (PM)  **Date**: 2026-03-27  **Status**: Draft
**核心命题**: 把 9 个模块从独立工具集变成一个有生命力的有机体

---

## 设计哲学

在研究了 OpenAI Operator、Anthropic Claude Computer Use、Microsoft Copilot Agents、Salesforce Agentforce、Superhuman AI、Notion 3.0 Agents、Lindy AI、飞书 Aily、钉钉悟空、Apple Intelligence 等全球领先实践后，我提炼出三个核心趋势：

1. **从被动响应到主动行动** -- AI 不再等用户提问，而是基于上下文信号自主触发（Apple Intelligence "Proactive Anticipation"、Superhuman Auto Drafts）
2. **从单点工具到跨应用编排** -- Agent 的价值不在单个功能，而在串联多个系统完成端到端工作流（Microsoft Copilot Cowork、Salesforce Agentforce 360 跨云编排、Notion Custom Agents 跨工具运行）
3. **从一个大 Agent 到专业化小 Agent 协作** -- 每个 Agent 有明确职责边界，通过共享上下文层协作（Lindy Agent Swarms、Salesforce 的 Topic-Action-Guardrail 架构）

**我们的设计原则**: 每个 Agent 必须至少串联 3 个模块，否则它只是一个"功能"而不是"Agent"。Agent 之间通过统一的上下文层（联系人图谱 + 时间线 + 承诺追踪）协作，而非直接耦合。

---

## 共享基础层：Unified Context Engine

在讨论具体 Agent 之前，需要先定义它们共享的"大脑"：

| 组件 | 数据来源 | 作用 |
|------|---------|------|
| 联系人图谱 | Gmail + WhatsApp + Calendar | 每个人的跨渠道身份合并、关系强度、最近互动 |
| 时间线 | 所有模块 | 某个人/项目的全部事件按时间排列 |
| 承诺状态机 | 出站邮件/WhatsApp + 跟进模块 | "我答应了X" / "我在等Y" 的实时状态 |
| 用户偏好模型 | Chat 历史 + 行为数据 | 写作风格、决策模式、时区偏好、餐厅口味 |

这一层借鉴了 **Microsoft Copilot 的 Work IQ 层**（跨 M365 的长期记忆）和 **Salesforce 的 Data Cloud Grounding**（让 Agent 接入实时数据而非静态快照）。

## 7 个 Agent 提案

### Agent 1: Radar（雷达扫描）-- 自动执行型

**优先级: P0**
**做什么**: 7x24 静默监控所有渠道，识别需要用户注意的"信号"，按紧急程度分级推送。不是简单的通知聚合，而是理解上下文后的智能过滤。

**串联模块**: Gmail + WhatsApp + Calendar + Tasks + Follow-ups + Contacts

**具体行为**:
- 识别"隐性紧急"邮件（例：投资人周五下午发邮件问进度，虽然语气平和但根据联系人关系判定为高优先级）
- 检测承诺到期预警（"你上周三答应 David 这周给报价，还剩 1 天"）
- 发现日历冲突的隐含问题（"明天 10am 会议的参会者 Sarah 上周 WhatsApp 说她这周在东京，时差 +1h"）
- 识别渠道切换信号（"某人在 WhatsApp 发了消息你没回，2 小时后又发了邮件 -- 可能比较急"）

**为什么有价值**: 创业者最大的痛苦不是没时间回消息，而是"不知道哪条消息最该先回"。当前 Daily Briefing 是每天一次的快照，Radar 是持续运行的实时守望。

**借鉴**: Superhuman 的 Auto Labels（自动分类邮件优先级）+ Apple Intelligence 的 Proactive Anticipation（基于模式预判用户需要什么）+ Notion Agent 的 24/7 自主运行模式。

**实现复杂度**: M（中等）-- 核心是在现有承诺追踪和联系人图谱基础上增加一个评分引擎和推送层。

---

### Agent 2: Ghostwriter（代笔人）-- 对话触发型

**优先级: P0**
**做什么**: 不只是"帮你写邮件"，而是基于完整上下文生成跨渠道的回复策略。你说"帮我回 David"，它知道 David 是谁、你们最近聊了什么、你答应了他什么、你的日程空余，然后生成回复。

**串联模块**: Gmail + WhatsApp + Calendar + Contacts + Tasks + Follow-ups

**具体行为**:
- 在 Chat 里说"回复 David 那封邮件"，Agent 自动拉取：David 的联系人画像、最近 5 次跨渠道互动、你对他的未履行承诺、你本周日历空余
- 生成回复草稿，包含对承诺的响应（"附件的报价如约发出"）和下一步安排（"下周二 3pm 我有空，要不要见面聊？"）
- 支持批量模式："帮我回完今天所有需要回复的邮件"，Agent 逐条生成，用户逐条确认或一键确认
- 自动选择最佳渠道（"David 上次用 WhatsApp 找你，回复建议走 WhatsApp 而不是邮件"）

**为什么有价值**: 当前的 AI 回复只看当前邮件内容。但真实世界中，回一封邮件需要记住你答应过什么、你的日程、对方的偏好。把这些上下文自动注入，回复质量从"能用"跃升到"像我本人写的"。

**借鉴**: Superhuman 的 Write with AI（适配个人写作风格 + 自动插入日程空余）+ Lindy 的 Agent（用自然语言描述任务，Agent 自动编排多步骤工作流）。

**实现复杂度**: M -- 核心是 context assembly pipeline，把多个模块的数据喂给 LLM prompt。

---

### Agent 3: Prep（会前准备官）-- 自动执行型

**优先级: P0**
**做什么**: 在每个日历事件前 30 分钟自动生成"作战简报"，不仅汇总参会者历史，还给出谈话策略建议。

**串联模块**: Calendar + Contacts + Gmail + WhatsApp + Tasks + Follow-ups + Restaurant

**具体行为**:
- 自动识别日历中"需要准备"的事件（不是所有事件都需要 -- 内部周会不需要，客户会议需要）
- 为每个参会者生成画像卡片：最近互动摘要、未完成的承诺、关系温度变化
- 如果是线下会议，自动推荐附近餐厅（"会议在 Tanjong Pagar，以下是 3 家适合商务午餐的选择"）
- 生成谈话要点建议："David 上周提到预算要到 Q3 才批，今天可能要讨论如何推进 bridging 方案"
- 检测潜在雷区："你答应 Sarah 的报告还没发，她也在这个会里，建议会前先处理"

**为什么有价值**: 已经完成的 P0"会议上下文卡片"是被动展示数据，Prep 是主动分析 + 行动建议。区别类似于"给你一叠文件"vs"帮你划好重点并告诉你该说什么"。

**借鉴**: Microsoft Copilot Cowork 的多步骤跨应用编排（会前准备天然是跨 Calendar + Mail + CRM 的多步骤任务）+ Salesforce Agentforce 的 Intelligent Context（基于非结构化数据理解上下文）。

**实现复杂度**: S-M -- 核心逻辑已有（会议上下文卡片），需要增加智能过滤（哪些会议需要准备）和策略生成层。

---

### Agent 4: Closer（闭环追踪器）-- 自动执行型

**优先级: P1**
**做什么**: 自动追踪所有"开环"事项（承诺、等待回复、会后待办），在接近超时时自动触发下一步动作链。不只是提醒你，而是帮你执行闭环。

**串联模块**: Follow-ups + Tasks + Gmail + WhatsApp + Calendar + Contacts

**具体行为**:
- 持续扫描所有"我在等 X 回复"的状态，如果超过预设阈值（默认 3 天），自动生成跟进草稿
- 跟进策略分级：第一次温和提醒 -> 第二次加入时间压力 -> 第三次建议换渠道（从邮件转 WhatsApp）或建议升级（"要不要直接约个电话？"）
- 会后自动提取 action items：会议结束后，基于日历事件 + 相关邮件线程，生成待办清单并分配到 Tasks
- 检测"死循环"：某个事项被反复推迟但从不完成，标记并建议用户决策（做/委派/杀掉）

**为什么有价值**: 现有的跟进模块告诉你"该跟了"，但执行仍然全靠用户。Closer 把"提醒 -> 起草 -> 发送"串成半自动链。对出差中的创业者，这意味着碎片时间里点一下"确认发送"就完成了一次跟进，而不是从零开始写。

**借鉴**: Lindy 的自主跟进 Agent（自动回复未读消息、跟进线索）+ 钉钉悟空的任务闭环追踪 + Notion Agent 的多步骤自主执行（最长运行 20 分钟完成复杂工作流）。

**实现复杂度**: M -- 跟进逻辑已有，需要增加自动升级策略和草稿生成。会后 action items 提取需要 LLM 处理会议相关文本。

---

### Agent 5: Weaver（关系织网者）-- 混合型（自动分析 + 对话查询）

**优先级: P1**
**做什么**: 主动维护你的人脉网络健康度。识别"正在冷却的重要关系"并建议行动，发现"隐藏的连接机会"。

**串联模块**: Contacts + Gmail + WhatsApp + Calendar + Restaurant

**具体行为**:
- 为每个联系人计算"关系温度"（基于互动频率、渠道多样性、最近互动时间、互动质量）
- 每周生成"关系健康报告"："以下 5 个重要联系人超过 30 天没互动，建议重新连接"
- 建议重连方式："你和 Emily 上次聊了她的新加坡扩张计划，可以分享这篇相关文章作为话题"
- 在 Chat 中支持查询："我下周去东京出差，有哪些联系人在东京？上次和他们聊了什么？"
- 会面安排建议："你和 Michael 都在新加坡，你俩 3 个月没见了，要不要下周约顿饭？" + 自动推荐餐厅

**为什么有价值**: 对创业者来说，人脉就是业务。但没有人有系统性维护关系的习惯。Weaver 把 CRM 级别的关系管理降维成一个不需要任何手动输入的自动系统。

**借鉴**: Salesforce Agentforce 的 CRM 自动化（关系追踪 + 下一步建议）+ Apple Intelligence 的 Proactive Suggestions（基于地理位置和行为模式的主动建议）+ 飞书的智能联系人管理。

**实现复杂度**: M-L -- 关系温度算法需要设计，联系人画像需要跨渠道数据聚合。但联系人图谱已有基础。

---

### Agent 6: Travel Brain（商旅大脑）-- 混合型

**优先级: P1**
**做什么**: 你的商旅过程中，自动编排与出行相关的所有准备、提醒和推荐。这是 Chief 的杀手级差异化 -- 没有任何竞品把商旅上下文深度融入通信助手。

**串联模块**: Calendar + Gmail + Contacts + Restaurant + Tasks + Daily Briefing + WhatsApp

**具体行为**:
- 检测日历中的出差信号（跨城市会议、航班确认邮件、酒店预订邮件），自动创建"出差时间线"
- 出差前 48h 自动准备："你在东京期间有 3 场会议，以下是每场的参会者简报 + 附近餐厅推荐 + 时区提醒"
- 出差期间 Daily Briefing 自动适配：按当地时区推送，高亮时差相关风险（"伦敦合伙人可能在你开会时发邮件，已标记优先级"）
- 在 Chat 中支持："帮我找下周东京行程里还没确认的事项" / "推荐一家适合请客户吃饭的日料"
- 回程后自动生成出差摘要：见了谁、承诺了什么、待跟进事项

**为什么有价值**: 这是我们定位中"商旅智能"差异化层的核心载体。出差是创业者最混乱的场景 -- 时区切换、密集会议、碎片时间。一个理解你整个出差上下文的 Agent 价值巨大。

**借鉴**: Google Project Mariner 的跨页面信息聚合能力 + Lindy 的时区感知调度 + Apple Intelligence 的地理感知上下文切换。但没有任何现有产品把这些整合进一个统一的商旅 Agent -- 这是白空间。

**实现复杂度**: L（大）-- 需要航班/酒店邮件解析、时区推理引擎、出差状态机。但核心价值极高，值得投入。

---

### Agent 7: Debrief（复盘助手）-- 对话触发型

**优先级: P2**
**做什么**: 在 Chat 中触发，帮用户对特定时间段或项目进行结构化复盘，生成洞察和行动建议。

**串联模块**: Gmail + WhatsApp + Calendar + Tasks + Follow-ups + Contacts

**具体行为**:
- "帮我复盘这周" -- 生成本周互动总结：见了谁、推进了什么、哪些事没做完、哪些承诺还开着
- "帮我复盘和 Acme Corp 的关系" -- 拉取所有与 Acme Corp 相关联系人的跨渠道互动历史，分析关系走势，建议下一步
- "帮我准备 board meeting" -- 汇总过去一个月的关键进展、关键对话、待解决问题，生成可直接用于汇报的结构化摘要
- 每月自动生成"月度回顾"推送：时间分配分析（会议时间/深度工作时间）、关系网络变化、承诺完成率

**为什么有价值**: 创业者永远在"做"，极少停下来"看"。Debrief 把散落在 9 个模块中的信息重新编织成叙事，帮用户看到自己看不到的模式。

**借鉴**: Notion AI Agent 的跨数据源汇总 + 分析能力（能自主运行 20 分钟完成复杂汇总任务）+ Microsoft Copilot 的 Work IQ（跨应用的长期记忆和模式识别）。

**实现复杂度**: M -- 主要是 LLM prompt engineering + 数据拉取编排。不需要新的数据管道，依赖已有模块的数据。

## Agent 交互矩阵

下表展示 7 个 Agent 如何通过共享上下文层协作：

| Agent | 给其他 Agent 的输出 | 从其他 Agent 接收的输入 |
|-------|-------------------|----------------------|
| **Radar** | 紧急信号 -> Ghostwriter（优先回复）, Closer（加速跟进） | Closer 的超时事件, Weaver 的关系降温警告 |
| **Ghostwriter** | 已发送回复 -> Closer（更新承诺状态）, Weaver（更新互动记录） | Radar 的优先级信号, Prep 的谈话要点 |
| **Prep** | 会前简报 -> Ghostwriter（会中可能需要的回复）, Closer（会后待办） | Weaver 的关系画像, Radar 的参会者相关警告 |
| **Closer** | 跟进结果 -> Weaver（互动事件）, Radar（状态更新） | Ghostwriter 的回复草稿, Prep 的会后 action items |
| **Weaver** | 关系洞察 -> Prep（参会者画像）, Travel Brain（目的地联系人） | 所有 Agent 的互动事件 |
| **Travel Brain** | 出差上下文 -> Prep（本地化会议准备）, Radar（时区感知优先级） | Weaver 的目的地联系人, Prep 的会议简报 |
| **Debrief** | 复盘洞察 -> Weaver（关系趋势）, Tasks（未完成项高亮） | 所有 Agent 的历史数据 |

## 优先级排序与落地节奏

### P0 -- 第一阶段（立即启动，4-6 周）

| Agent | 理由 | 工程量 |
|-------|------|--------|
| **Radar** | 把现有 Daily Briefing 从"每日一次"升级为"实时守望"，直接提升核心价值感知 | 2-3 周 |
| **Ghostwriter** | 把现有 AI 回复从"单邮件"升级为"上下文感知"，用户立刻感受到"这个 AI 真的了解我" | 2-3 周 |
| **Prep** | 会议上下文卡片的自然延伸，投入小产出大 | 1-2 周 |

**P0 的核心命题**: 让用户从"我有 9 个工具"变成"我有一个懂我的助手"。

### P1 -- 第二阶段（P0 上线后 4-6 周）

| Agent | 理由 | 工程量 |
|-------|------|--------|
| **Closer** | 把跟进从"提醒"升级为"半自动执行"，直接节省用户时间 | 3-4 周 |
| **Weaver** | 联系人图谱已有，加上关系温度算法后价值质变 | 3-4 周 |
| **Travel Brain** | 差异化杀手锏，但需要更多数据解析工作 | 4-6 周 |

**P1 的核心命题**: 让用户从"助手提醒我做事"变成"助手帮我做事"。

### P2 -- 第三阶段（P1 验证后）

| Agent | 理由 | 工程量 |
|-------|------|--------|
| **Debrief** | 高价值但非紧急，需要足够的历史数据积累才有意义 | 2-3 周 |

**P2 的核心命题**: 让用户从"助手帮我做事"变成"助手帮我思考"。

## 自动执行 vs 对话触发 分类

| 类型 | Agent | 触发方式 | 用户介入程度 |
|------|-------|---------|------------|
| **自动执行型** | Radar | 持续后台运行，推送通知 | 只需看/忽略 |
| **自动执行型** | Prep | 日历事件前 30min 自动触发 | 只需阅读简报 |
| **自动执行型** | Closer | 超时阈值触发 | 确认/修改草稿后发送 |
| **对话触发型** | Ghostwriter | 在 Chat 中"帮我回复 X" | 审核草稿后确认 |
| **对话触发型** | Debrief | 在 Chat 中"帮我复盘 X" | 阅读并追问 |
| **混合型** | Weaver | 每周自动报告 + Chat 中可查询 | 看报告 / 主动查询 |
| **混合型** | Travel Brain | 检测出差自动激活 + Chat 可查询 | 看准备清单 / 主动查询 |

## 关键设计决策

### 决策 1: Human-in-the-loop 原则

所有对外发送的动作（邮件、WhatsApp消息）必须经过用户确认。Agent 可以自动准备，但不能自动发送。

**理由**: 对创业者来说，每条对外消息都代表个人品牌。自动发送的风险远大于收益。这与 Anthropic Claude Computer Use 的设计哲学一致 -- "Agent 做准备，人做决定"。

**唯一例外**: 用户可以为特定场景设置"自动模式"（如：对所有会议邀请自动回复"已确认"）。

### 决策 2: Agent 不互相直接调用

Agent 之间通过共享上下文层通信，而非直接调用。Radar 不会调用 Ghostwriter 的 API；它把信号写入上下文层，Ghostwriter 在被触发时读取。

**理由**: 避免 Agent 之间形成复杂依赖链导致调试噩梦。借鉴 Salesforce Agentforce 的 Topic-Action-Guardrail 分离架构。

### 决策 3: 先做 Unified Context Engine，再做 Agent

Agent 的质量完全取决于上下文层的质量。建议在开发任何 Agent 之前，先用 1-2 周强化联系人图谱和承诺状态机。

**理由**: 一个拥有完美上下文的简单 Agent，远比一个上下文残缺的复杂 Agent 有价值。

## 风险与缓解

| 风险 | 可能性 | 影响 | 缓解方案 |
|------|--------|------|---------|
| Agent 产生幻觉/错误上下文 | 高 | 高 | 所有 Agent 输出标注信息来源；对外消息必须人工确认 |
| 用户被过多推送淹没 | 中 | 高 | Radar 严格限制推送频率（默认最多 5 次/天），支持用户调节灵敏度 |
| LLM 成本随 Agent 数量线性增长 | 高 | 中 | 分级模型策略：Radar 用小模型做信号检测，Ghostwriter 用大模型做生成 |
| Agent 之间上下文不一致 | 中 | 中 | 统一 Context Engine，所有 Agent 读同一份数据快照 |
| 商旅邮件解析准确率不足 | 中 | 中 | Travel Brain 用规则引擎 + LLM 双层解析；解析失败时优雅降级为手动输入 |
