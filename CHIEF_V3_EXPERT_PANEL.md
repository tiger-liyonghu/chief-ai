# Chief v3.0 Expert Panel Discussion
## 8位专家三轮讨论 | 2026-03-30

---

## 参与专家

1. **McKinsey Partner** — 战略定位、市场差异化
2. **Executive Secretary (20年资深)** — 真正的顶级秘书做什么、不成文规则
3. **System Architect** — 统一数据架构、上下文图谱、实时系统
4. **LLM Expert** — prompt工程、多模型路由、成本优化、精度
5. **AI Agent Expert** — 自主Agent、工具编排、记忆系统
6. **Product Expert** — 熟悉Chief架构（32工具、三维定位、Supabase、95%精度）
7. **Linguist** — 多语言（中/英/马来/Singlish）、文化差异、沟通语调
8. **EQ Expert** — 情商、关系动态、信任构建、何时说何时沉默

---

## 第一轮：各自愿景

### McKinsey Partner
- Chief的问题不是技术问题，是战略问题
- 32个工具组成9个能力圈，但能力不等于战略
- 核心洞察：**高管的效能取决于承诺的质量**
- 每个会议是时间的承诺，每次出差是到场的承诺，每个家庭活动是优先级的承诺
- Chief应该是**承诺图谱引擎**，不是工具分发器
- 护城河不是32个工具，是积累的承诺上下文——用了6个月后无法切换
- 95%提取精度是基础，防御价值在推理层："你答应了投资人X周五前发board update，但你周四飞Jakarta，草稿还没开始。你的选项是..."

### Executive Secretary (20年资深)
- 软件人总是搞错秘书做什么：**我们不管日历，我们管老板的声誉**
- 真正的秘书不会查5个数据库回答"明天什么安排"——她已经知道了
- 周一早上是深度工作时间，见董事长前要留30分钟buffer，老婆生日晚餐不能改哪怕总理打电话
- Chief需要的不是更多工具，是**判断何时说话、何时沉默**
- 老板说"取消明天所有会议"——初级秘书全取消，资深秘书20分钟后回电话说"下午清了，但10点跟监管的保留了，因为改期会发错误信号"
- 这是Chief需要达到的水平

### System Architect
- 4-5轮工具调用是架构臭味，不是bug
- 32个工具按**功能**组织（日历、联系人、出差），应该按**实体**组织
- 提议**统一上下文图谱**：三个核心实体`persons`、`commitments`、`contexts`
- 当前每个工具维护自己的状态——出差工具知道城市，日历工具知道事件，联系人工具知道人——但没人知道"Jakarta周四的晚餐是跟你答应发term sheet的投资人，而你老婆父母同一天到"
- 实现路径：在现有Supabase表上建**物化视图层**，不重写32个工具
- 4-5轮变1轮，不是工具更快，而是调用更少

### LLM Expert
- 4-5轮问题主要是LLM推理问题——DeepSeek被要求在对工具返回内容和关联一无所知的情况下规划工具使用策略
- 两层修复：
  1. **工具合并**：32个太多，研究表明超过15-20个工具准确率下降。合并为8-10个**复合工具**
  2. **模型路由**："几点的航班"用便宜快速模型，"要不要因为Jakarta行程改board meeting"用完整推理链
- 3-tier路由：Tier 1（查询<100ms最便宜模型）、Tier 2（单步推理<500ms DeepSeek）、Tier 3（多步规划<3s最强模型）

### AI Agent Expert
- 当前是**工具型聊天机器人**，不是Agent。区别很大
- 聊天机器人响应查询，Agent追求目标
- 缺失的关键原语：**目标循环**
- 现在：用户说"提醒我周五前发deck给James" → 存提醒 → 周五触发
- Agent做法：持续监控承诺状态。周三发现drafts里没有deck。周四早上看到三个背靠背会议没时间准备。主动提醒："你承诺周五给James发deck，明天没有准备时间。要今晚block 90分钟还是推deadline？"
- 记忆架构三层：情景记忆（发生了什么）、语义记忆（什么是真的）、工作记忆（现在什么重要）
- 当前Supabase表是情景记忆，联系人/偏好是碎片化的语义记忆，几乎没有工作记忆

### Product Expert
- 32个工具不是随意设计的，来自真实用户场景
- 95%精度来之不易且脆弱，依赖精心调优的prompt
- 4-5轮问题严重性不同："明天什么安排"确实浪费，但"帮我规划Jakarta行程"多轮是正确的
- 过度统一的危险：建一个God Object返回所有东西，LLM要过滤3000 token上下文blob会产生幻觉
- 建议：保留32个工具作为实现层，引入**预取编排器**，根据消息分类预加载相关上下文包

### Linguist
- 没人提到Chief运营在全球语言最复杂的环境之一
- 新加坡创始人的WhatsApp消息包含英语、普通话、马来语、福建话语气词、Singlish语法——有时在同一句话里
- "Can"=是。"Can lah"=勉强是。"Can meh?"=我怀疑。"Confirm can"=绝对是。这不是同义词——它们携带不同的置信度
- "I try to come lah"不是确认出席——是带社交压力的软性也许
- 95%精度在这些模糊承诺上可能降到70%
- 回复生成同样关键：给投资人的邮件和给老朋友的WhatsApp必须code-switch
- 需要**逐关系的语域模型**——不只是"什么语言"，而是"什么正式度、什么语气词、什么文化标记"

### EQ Expert
- 挑战基本假设：Chief的工作不是让高管更**高效**，而是让高管在关系中更**有效**
- 有时有效意味着故意低效
- 关系时刻不是生产力的中断——它们就是产品本身
- Chief应该像CRM追踪pipeline一样追踪关系健康——但用情感粒度，不只是交易频率
- 最有情商的能力：**预判高管应该做但会忘记的事**
- "你跟联合创始人说会考虑她的提案。已经5天了。她可能在想。"
- "你的投资人James提到他儿子在申请NUS。你在招生办有联系人。"
- 这些是**关系动作**——世界级秘书本能做的事

## 第二轮：交叉质询

### McKinsey Partner回应Architect和Product Expert
- 两人从相反方向解决同一问题，都不完全对
- 战略问题：Chief应该**拥有vs借用**什么数据？
- 如果Chief试图成为日历、联系人、出差的记录系统，同时跟Google Calendar、Apple Contacts、TripIt竞争——自杀
- Chief应该只拥有一样东西：**承诺图谱**。其他一切从规范来源通过API拉取
- 反驳Secretary的"沉默"建议：前90天Chief必须在每次交互中证明价值，沉默是建立信任后才有的奢侈

### Executive Secretary回应McKinsey Partner和EQ Expert
- McKinsey Partner说沉默是奢侈。20年经验告诉我：**没有什么比不必要的噪音更快地摧毁信任**
- 第三个CEO解雇了前任秘书因为她发太多"just checking in"消息
- WhatsApp渠道是神圣的——老板的私人空间
- 但强烈同意EQ Expert的关系时刻。曾提醒CEO投资人女儿大学毕业，CEO发了祝贺信，两周后谈判关闭
- 反对"关系健康仪表盘"。量化关系感觉临床。"关系评分62/100"是反乌托邦
- 信号应该微妙——温和的提醒，不是记分卡

### System Architect回应LLM Expert和Agent Expert
- 复合工具有隐藏扇出的调试噩梦风险
- 3-tier路由需要反馈循环。"Is James coming Thursday?"看起来像简单查询但可能需要承诺分析
- 工作记忆是最难的工程问题。消息间工作记忆会过时，过时的工作记忆比没有更糟
- 提议：工作记忆用**cron驱动刷新**（每30分钟），不是事件驱动

### LLM Expert回应Linguist和Product Expert
- Linguist的置信度问题需要形式化
- 提议**4级承诺置信度模型**：Confirmed(>90%)、Likely(70-90%)、Tentative(40-70%)、Unlikely(<40%)
- God Object的解决方案不是保持孤立——是**层次化结构上下文**
- LLM先收到摘要层（200 token："明天4个会议，2个未了承诺，1个出差冲突"）
- 需要细节时定向查询

### AI Agent Expert回应Secretary和Architect
- Secretary的沉默直觉完全正确——对应推送通知系统中的**通知疲劳崩溃**
- 解决方案不是"更少主动"——是"在恰好正确的时刻主动"
- 需要**主动性模型**两个轴：紧急度（时间敏感性）和关系敏感度（错过的损害）
- 对Architect的过时工作记忆：提议**事件触发刷新+防抖**（60秒防抖）比纯cron更好

### Linguist回应LLM Expert和EQ Expert
- 4级置信度模型缺少文化维度
- "Let me check my schedule and get back to you"从新加坡高管=60%可能拒绝，从澳洲expatriate=80%真的会查
- 承诺置信度必须**按人校准，不是按短语**
- 关系漂移的语言信号：**回复时间不对称**和**消息长度缩短**是更强的信号

### EQ Expert回应Secretary和McKinsey Partner
- 不提议量化评分，提议**模式识别呈现为自然语言**
- "你和James以前每月见一次。上次是10月。"这是贴心朋友会说的话
- 挑战McKinsey Partner的"承诺图谱引擎"框架：承诺只是一个维度
- 还有**债务**（欠的人情）、**投资**（没有直接回报的关系建设）、**信号**（传达优先级和尊重的行为）

## 第三轮：共识

### 全部8位专家一致同意的5点

1. **32工具孤岛架构必须进化**——4-5轮工具链是碎片化数据的症状
2. **人+承诺+场景是正确的核心三角**——但"承诺"应扩展（加入债务、投资、信号）
3. **主动性是差异化因素，但通知纪律是生存线**——紧急度/关系敏感度矩阵决定何时推送
4. **语言和文化智能不是功能——是基础设施**
5. **Chief必须先赢得信任再变聪明**——先证明基础能力可靠

### 有建设性的分歧

- **重写vs演进**：共识是**先建预取编排器作为统一图谱的试验场**
- **拥有vs借用**：**拥有承诺图谱，缓存其他一切并标注时效**
- **主动vs沉默**：**从每日一条结构化简报开始，通过准确性赢得更高频率的权利**

## 共识文档

### I. 五个不可协商的设计原则

1. **以承诺为中心的数据模型** — 每个功能都追溯到人+承诺+场景三角形
2. **置信度优于确定性** — 永远不把不确定信息当事实呈现，4级置信度
3. **赢得说话的权利** — 默认每日结构化简报，主动告警只用于时间敏感冲突和关系关键时刻
4. **文化智能作为基础设施** — 语域切换、文化日历、按人行为校准不是功能而是底层
5. **可逆决策优于最优决策** — 现阶段快速改方向的能力比选理论最优架构更有价值

### II. 统一架构

```
WhatsApp (Baileys)
    ↓
Message Classifier (意图+紧急度+语域)
    ↓
Model Router (3-tier: T1查询 / T2推理 / T3规划)
    ↓
Context Resolution Layer (预取编排器)
  → 在LLM调用前解析 Person+Commitment+Context
  → 从三个数据源水合:
    Commitment Store (拥有，规范)
    Person Graph (拥有+增强)
    Context Cache (外部，TTL)
    ↓
Supabase (统一)
  commitments (拥有，规范)
  persons (拥有+增强)
  context_cache (外部，TTL)
  interaction_log (只追加)
  relationship_signals (派生)
    ↓
Background Agents
  承诺监控 (cron 30m)
  关系漂移检测
  每日简报编译
  上下文缓存刷新
```

**32个工具保留为实现层**，但LLM看到的是8个复合工具：
`resolve_day`, `resolve_person`, `resolve_commitment`, `resolve_trip`, `draft_response`, `suggest_action`, `update_commitment`, `search_context`

### III. 三个最高影响力变更（优先实施）

1. **Context Resolution Layer (第1-3周)** — 预取编排器，5类消息分类，预加载数据包。工具轮次从4.2降到1.5，P95延迟降低60-70%
2. **承诺置信度模型 (第2-4周)** — 4级置信度+东南亚对冲模式+按人历史校准
3. **每日简报Agent (第3-5周)** — 7:30am WhatsApp推送，含承诺注释的日程+需回复消息+一条关系提醒。信任构建机制

### IV. 明确不做的事（反模式）

1. **现在不重写数据库Schema** — 让Context Resolution Layer先证明哪些数据模式重要
2. **不建关系健康评分** — 用自然语言观察代替数字评分
3. **不再添加工具** — v3的路径是更深集成现有能力，不是更多能力
4. **不为速度牺牲准确** — 错误的100ms答案比正确的2秒答案更差
5. **不代发任何消息** — 所有外发必须人工确认，无例外
